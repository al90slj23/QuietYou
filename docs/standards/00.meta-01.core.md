## 🏗️ 文档体系架构

### 文档分类

| 类型 | 位置 | 读者 | 特点 |
|------|------|------|------|
| 完整版 | `docs/standards/` | 人类 + AI（需要时） | 原则性、详细说明 |
| 精简版 | `.kiro/steering/` | AI（自动加载） | 核心规则、完整版引用 |

### 双版本同步机制

精简版与完整版**文件名完全一致**，一一对应：

```
docs/standards/                    .kiro/steering/
├── 00.meta-01.core.md        ←→   00.meta-01.core.md
├── 01.arch-01.core.md        ←→   01.arch-01.core.md
├── 01.arch-02.structure.md   ←→   01.arch-02.structure.md
├── 02.backend-01.api.md      ←→   02.backend-01.api.md
├── 03.frontend-01.structure.md ←→ 03.frontend-01.structure.md
├── 04.quality-01.naming.md   ←→   04.quality-01.naming.md
├── 04.quality-02.git.md      ←→   04.quality-02.git.md
├── 04.quality-03.header.md   ←→   04.quality-03.header.md
└── 09.tool-01.gosh.md        ←→   09.tool-01.gosh.md
```

### 更新规范

当说"更新到 docs"时，根据内容类型执行不同流程：

#### 开发规范更新（docs/standards/）

**两步同步流程：**

1. **更新完整版** `docs/standards/xx.xxx-xx.xxx.md`
   - 包含完整说明、示例、原理
   - 人类可读的详细文档

2. **同步精简版** `.kiro/steering/xx.xxx-xx.xxx.md`
   - 提取核心规则
   - 顶部添加完整版引用
   - AI 自动加载

#### 产品文档更新（docs/product/）

**单步流程：**
- 直接更新 `docs/product/` 对应文件
- 无需同步到 steering（产品文档不需要 AI 自动加载）

#### 新增规范文档

1. 确定大类编号（00-09）
2. 按命名格式创建完整版
3. 创建同名精简版
4. 更新本文件的文件对应关系表

### 精简版格式要求

每个精简版文件顶部必须包含完整版引用：

```markdown
# 标题（精简版）

> 完整版: `docs/standards/xx.xxx-xx.xxx.md`

[精简内容...]
```
